# 视频尾帧约束与质量评估体系：问题分析及架构改进方案

日期：2026-07-24  
范围：HappyHorse 图生视频的尾帧连续性，以及视频生成后的系统评分、视觉模型评分和人工审核  
说明：本文只给出分析与方案，不包含代码实现。

## 1. 结论摘要

当前两个问题表面上分别是“HappyHorse 不听尾帧提示”和“评分不稳定”，本质上却来自同一个架构缺口：

> 系统把模型不具备的硬能力，当成了 Prompt 可以补齐的软能力；又把不确定的模型判断和不稳定的基础设施调用，当成了确定的质量结论。

因此，不建议继续单纯增加提示词长度，也不建议只调低视觉模型的通过阈值。更合理的方向是：

1. 把“必须精确到达尾帧”定义成生成任务的能力要求，在调度层选择真正支持首尾帧输入的模型；HappyHorse 只承接尾帧允许为软目标的任务。
2. 把现有自然语言大 Prompt 改造成“结构化镜头合同 → 模型专属 Prompt 编译器”，限制长度、去重、消除冲突，并按优先级输出。
3. 将评分拆为“技术可用性、可测量事实、语义质量、风险建议”四层；评估失败必须显示为“未评估”，不能显示为 0 分。
4. 视觉模型不再独占最终通过权。硬规则由确定性检查器判定，主观项由视觉模型评分；只有高置信度、与合同直接相关的严重问题才能自动阻断。
5. 用真实人工标注样本校准阈值和误杀率。在校准完成前，视频视觉评分应以排序和风险提示为主，不宜把绝对分数当作发布许可。

## 2. 当前实现说明与直接证据

### 2.1 HappyHorse 的尾帧不是硬输入

当前能力声明已经明确：

- 支持首帧图片输入。
- 不支持尾帧图片输入。
- 尾帧模式是 `soft_prompt_target`。

实际提交到上游的媒体内容也只有一张 `first_frame`。虽然业务方法要求存在已审核尾帧，但尾帧 URL 只用于编译 Prompt 和生成后比较，并没有作为模型输入。

这意味着：只要继续使用 HappyHorse 当前接口，系统就无法“保证”末帧到达指定图像，只能提高到达概率。任何把它描述成硬约束的产品承诺都会高于底层模型真实能力。

### 2.2 当前 Prompt 已经很强，但缺少编译治理

当前视频 Prompt 已经包含：

- 强制终态标题和尾帧合同。
- 从尾帧倒推动作时间。
- 要求主动作提前完成并在末尾稳定停留。
- 禁止中途结束、越过终态、切镜、溶解、贴帧。
- 起始状态、结束状态、运动路径、单镜头合同、运动检查点。
- 叙事边界、角色和产品锁、机位继承、负面规则。
- 上一次视觉检查产生的重试指令。

因此，问题不能简单归因于“没有强调尾帧”。更值得注意的是以下工程风险：

1. **同一要求被多种表达重复描述。** 尾帧蓝图、尾帧合同、叙事结果、检查点和重试指令可能描述同一目标，也可能局部冲突。
2. **把原始长文本和 JSON 直接拼入 Prompt。** 模型需要自己从大量描述中重新判断优先级。
3. **缺少统一的字符预算。** Prompt 在提交前会被截到 5000 字符，但编译阶段没有对每个区块分配预算。排在后面的检查点、锚点锁和机位规则存在被截断或表达不完整的风险。
4. **职责过多。** 一个 Prompt 同时承担叙事防越界、视觉身份、空间连续性、镜头运动、声音、文字禁令、终态到达和重试修复，注意力天然会被稀释。
5. **重试是在原长 Prompt 上继续叠加。** 如果不先删除已失效或低优先级描述，重试指令只会让合同更拥挤。

所以，“Prompt 太长太乱”是问题的一部分，但不是唯一原因。更根本的原因是缺少 Prompt 中间表示、优先级、预算和模型能力适配。

### 2.3 评分失败被错误表达成质量 0 分

代码中存在多条评分路径，语义并不统一：

- 一条本地启发式路径主要根据任务状态、是否有视频 URL、是否有缩略图、时长是否接近目标来加分，还加入了基于 ID 的微小随机区分。这实际上是“交付完整度/技术健康度”，不是视觉质量。
- Python 视频质检接口在下载、请求模型、解析 JSON 等任一步失败时，把总分和四个维度全部写成 0。
- One Prompt Video 的技术评估失败报告也把各维度暂填为 0，虽然新版界面已开始把这类结果显示为破折号并标记“分析不可用”，但数据模型里仍保留了容易被误用的 0。
- 缺失模型字段会被数字归一化函数默认为 0，从而把“模型未返回”变成“画面极差”。

这会造成三类后果：

1. 用户误以为视频质量为 0，实际上可能只是超时、限流、视频下载失败、FFmpeg 失败或 JSON 解析失败。
2. 排序、平均分、自动筛选等下游逻辑把技术故障当作内容质量参与计算。
3. 团队无法从数据上区分“模型真的判差”与“评估系统自身崩溃”，进而无法正确优化。

### 2.4 视觉模型通过率低不是单纯阈值问题

当前视频质量判断同时依赖：

- 视觉模型自己的 `passed=true`。
- identity、layout、prompt alignment、continuity 等基础分阈值。
- first-frame consistency、checkpoint order、single-take 等视频专属阈值。
- 元数据必须无问题。
- 另一个端帧检查还要求模型判定通过且尾帧相似度至少为 0.8。

这里存在几个系统性问题：

1. **多个门禁串联会放大误杀率。** 即使每个门禁单独有 90% 的正确放行率，七个近似独立的门禁串联后，整体放行率也只有约 48%。
2. **视觉模型既打分又给最终否决。** 模型的一次保守判断会覆盖其他高分证据。
3. **一套样本承担太多任务。** 五张视频采样帧同时用于判断身份、动作、连续性、切镜、终态和文字，信息不足时模型仍被要求给确定答案。
4. **参考图片可能重复。** 首帧和尾帧已单独加入评估内容，选中参考图中又可能再次包含它们，增加角色混淆和上下文负担。
5. **终态标准过于绝对。** 当前端帧检查要求身份、主体状态、构图、机位、布局、灯光和动作结果全部可见匹配。对只有首帧硬输入的生成模型而言，这更像验收一个首尾帧模型。
6. **缺少“不确定/不可判断”的完整通道。** 尽管部分 Prompt 要求低置信度问题不能否决，但最终数据归一化和串联门禁仍容易把缺失或不确定压成失败。
7. **没有基于人工样本的阈值校准证据。** 65、60、80 等阈值是工程规则，但尚不能说明它们与人工可用判断具有足够相关性。

## 3. 问题一：尾帧约束的目标架构

### 3.1 先把“尾帧要求”从 Prompt 问题提升为能力问题

建议为每个视频片段明确一种终态等级：

| 等级 | 业务含义 | 推荐执行方式 |
|---|---|---|
| Hard Exact | 构图、主体姿态、物体位置等必须接近指定尾帧 | 只路由到支持真实首尾帧输入的模型或工作流 |
| Hard Semantic | 必须完成某个可见结果，但允许构图和细节变化 | 可用首帧模型，但必须多候选择优并做终态检查 |
| Soft Directional | 尾帧只是期望方向，允许未完全到达 | HappyHorse 单首帧生成 |
| Editorial | 只需为后续剪辑提供可连接状态 | 以剪辑接点、运动方向和画面稳定性为主 |

用户审核尾帧时，系统应明确告诉用户当前片段属于哪一级。若业务选择 Hard Exact，而所选模型不支持尾帧硬输入，应该在提交前提示能力不匹配并自动换路由，而不是生成后反复扣费重试。

### 3.2 建立模型能力注册表和能力驱动路由

模型注册信息不应只记录名字和价格，还应包含：

- 是否支持首帧。
- 是否支持真实尾帧。
- 是否支持多参考图。
- 最大 Prompt 长度和推荐长度。
- 推荐时长范围。
- 是否支持镜头运动、音频和负面提示。
- 终态控制可靠性等级。
- 支持的宽高比和分辨率。
- 已知失败模式及建议评估方式。

规划层输出的是需求，调度层根据能力选模型。这样未来替换 HappyHorse、接入新的首尾帧模型或 RunningHub 工作流时，不需要重写业务合同。

### 3.3 使用“镜头合同中间表示”，不要直接堆自然语言

建议在生成前先形成一个模型无关的镜头合同，至少包含：

- 起始可见状态。
- 必须到达的最小终态。
- 终态中不可妥协的 1–3 个可观察事实。
- 连续动作路径。
- 最多 3 个中间检查点。
- 必须保持的身份、物体和空间关系。
- 允许变化的内容。
- 禁止事项。
- 末尾预留稳定时间。

这里最重要的是“最小终态”。不要把整张尾帧的所有视觉细节都当成同等重要的硬要求。例如真正影响下一段衔接的可能只有人物位置、朝向、手中物体和机位轴线；背景装饰和细微灯光可以是软要求。要求越少、越可见、越可执行，模型到达概率越高。

### 3.4 为 HappyHorse 单独编译短而有优先级的 Prompt

建议 Prompt 固定为以下顺序：

1. **模式声明**：首帧单镜头图生视频。
2. **唯一主目标**：一句话说明在第几秒前到达什么终态。
3. **终态验收清单**：最多 3 条，每条都是画面中可直接观察的事实。
4. **动作时间表**：开始、主要动作完成、稳定停留三个时间段。
5. **连续运动检查点**：最多 3 个。
6. **保持项**：身份、产品实例、空间轴线等最多 5 项。
7. **关键禁令**：只保留与当前片段实际风险相关的禁令。
8. **本轮重试增量**：最多 2 条，并说明其优先级高于上一轮非硬性细节。

编译器还应执行：

- 重复语义合并。
- 相互矛盾检测。
- 运动过程和静态终态分离。
- 每区块字符预算。
- 总长度预检。
- 截断前告警，禁止静默截断。
- Prompt 指纹和版本记录，便于回归分析。

### 3.5 当必须继续使用 HappyHorse 时，提高概率而非承诺保证

可组合使用以下策略：

1. **倒排动作预算。** 主动作在视频总时长的 60%–75% 前完成，剩余时间只用于减速和稳定终态。
2. **降低单段状态跨度。** 如果从首帧到尾帧包含人物大位移、转身、拿取物体和大幅运镜，应在规划阶段拆段，而不是让 Prompt 同时承载多个高风险变化。
3. **多候选并行。** 每次生成多个候选，以终态达成度排序，优先选择最接近目标者，而不是对单候选无限重试。
4. **失败差量重试。** 重试只写“当前末帧与目标的最大两个差异”，不要再次附加完整诊断历史。
5. **短桥接片段。** 若主片段运动很好但末态略有偏差，可用一个支持首尾帧的短桥接模型补连接；应作为明确的编辑资产，不在原片段内贴静帧或伪造到达。
6. **供应商切换。** 对 Hard Exact 任务，这是唯一真正能把软约束升级为硬约束的方式。

不建议恢复末尾贴图、冻结或 dissolve 强行接尾帧。它只能制造表面一致，通常会带来速度突变、人物形变、假停顿和明显的剪辑痕迹。

## 4. 问题二：评分体系的目标架构

### 4.1 统一评分语义，先消灭“双评分体系”

建议停止使用含糊的“系统评分”和“模型评分”，改为四类明确结果：

| 层级 | 回答的问题 | 典型输出 |
|---|---|---|
| 技术检查 | 文件能否下载、解码、播放，时长和尺寸是否有效 | 通过 / 失败 / 未检查 |
| 确定性规则 | 时长、黑帧、静帧、重复帧、音轨、分辨率等是否合规 | 事实与错误码 |
| 视觉语义评估 | 身份、构图、动作、终态、连续性是否符合合同 | 分维度分数、置信度、证据 |
| 发布风险 | 是否需要人工看、是否建议重试、是否阻断 | 自动通过 / 建议复核 / 必须复核 / 基础设施异常 |

现有基于 URL、缩略图、状态和时长的本地分数，可以保留但应更名为“交付健康度”，绝不能与视觉分数求平均或用于判断画面好坏。

### 4.2 分数必须可空，评估状态必须一等公民

建议每次评估至少区分：

- Pending：尚未评估。
- Running：评估中。
- Completed：成功得到有效判断。
- Partial：部分维度可判断。
- Unavailable：服务暂不可用。
- Invalid Input：视频或参考资料不可读。
- Needs Re-evaluation：结果格式异常或置信度不足。

只有 Completed 或明确可用的 Partial 才能产生分数。其他状态的分数应为空，不是 0。UI、排序、均值、自动筛选和数据库统计都必须遵守这一语义。

### 4.3 硬门禁与软评分分开

建议硬门禁仅保留少量高确定性条件，例如：

- 视频文件不可播放。
- 合同明确禁止的错误品牌文字被高置信度检测到。
- 角色或产品实例数明显错误。
- 存在明确跳切，而合同要求单镜头。
- Hard Semantic 终态的核心动作结果完全未发生。

身份相似度、构图美感、轻微动作僵硬、灯光差异、末帧相似度不足等应进入软评分或“建议人工复核”，不要全部自动否决。

最终决策不应是简单的“所有阈值都通过”。更合适的决策顺序是：

1. 技术检查失败：修复评估基础设施或媒体，不评价内容。
2. 存在高置信度硬失败：自动重试或阻断。
3. 无硬失败且综合质量较高：自动通过。
4. 分数处于灰区或模型低置信度：人工抽检。
5. 多候选都可用：用成对比较或排序模型选择更优者。

### 4.4 将视觉评估拆成专职评估器

不要让一个视觉模型调用同时回答所有问题。建议拆为：

- 首帧一致性评估器。
- 身份和实例连续性评估器。
- 动作与单镜头评估器。
- 尾帧核心终态评估器。
- 文字、Logo 和 UI 评估器。
- 技术元数据检查器。

并非每个片段都运行全部评估器。应根据镜头合同和风险标签选择需要的检查项。例如没有文字要求的纯人物镜头，不应让文字检查影响通过；没有大幅运动的镜头，也不需要复杂的 checkpoint 顺序门禁。

### 4.5 视觉模型 Prompt 应改成“证据—判断—置信度”

每个评估器只接收完成任务所需的最小内容：

- 当前输出。
- 与本检查直接相关的参考图。
- 最小可观察合同。
- 明确的“不适用”和“无法判断”选项。

每个问题必须返回：

- 观察到了什么。
- 证据位于哪一帧、哪个区域。
- 对应哪条合同。
- 置信度。
- 严重程度。
- 建议动作。

模型不得仅凭低分否决；只有“适用、证据明确、置信度达到门槛、严重程度为硬失败”四项同时满足，才进入自动阻断。

### 4.6 视频采样应与问题匹配

固定抽五帧适合做粗略筛查，但不足以稳定判断快速跳切、动作顺序和短暂异常。建议分层采样：

- 技术层用 FFprobe、帧差、黑帧和冻结检测。
- 首尾一致性读取真正的首个可解码帧和最后稳定帧。
- 动作层先做均匀采样；发现高变化区间后再局部加密采样。
- 文字和 Logo 检查选取清晰度最高、目标区域最大的帧。
- 终态评估使用最后一小段的多帧稳定窗口，而不是单独一张末帧。

“末尾稳定窗口”尤其重要：若最后一帧恰逢运动模糊、编码尾部或淡出，单帧判断会误杀一个整体可用的视频。

## 5. 可靠性与可观测性改进

评分系统需要把内容失败与系统失败分别统计。建议记录：

- 视频下载成功率和耗时。
- FFmpeg/FFprobe 成功率。
- 视觉模型请求成功率、超时率、限流率。
- JSON 合法率和字段缺失率。
- 每个评估器的重试次数。
- 评估模型版本、Prompt 版本、规则版本。
- 单片段输入图片数量和上下文大小。
- 自动通过率、人工推翻率、误杀率、漏放率。
- 不同模型、镜头类型和终态等级下的通过率。

同时需要：

- 对同一媒体和同一评估版本做结果缓存，避免重复付费。
- 使用幂等任务和独立评估队列，生成成功不应因评估超时而变成生成失败。
- 对限流、网络错误、解析错误做分类重试。
- JSON 校验失败时先进行格式修复或重新请求，不能直接记 0。
- 设置熔断和降级：评估服务连续异常时允许视频进入“待人工抽检”，而不是让全链路堵死。

## 6. 人工校准与上线策略

视觉评分体系没有人工校准集，就无法知道“严格”到底是质量保障还是系统性误杀。建议建立一个小而真实的黄金集：

- 覆盖不同人物、产品、游戏 UI、运镜和时长。
- 每类至少包含明确通过、明确失败和争议样本。
- 每个样本由至少两名人工按统一 rubric 标注。
- 对尾帧分别标注核心状态达成、构图接近、身份一致和可剪辑性。

上线目标不应先追求“模型平均分更高”，而应追求：

- 技术失败不再显示 0 分。
- 评估服务失败不触发重新生成。
- 自动阻断结果的人工认同率足够高。
- 人工可接受视频被自动拒绝的比例持续下降。
- 多候选择优与人工首选的一致率持续上升。
- HappyHorse 在 Hard Semantic 任务上的终态达成率可量化。
- Hard Exact 任务不再错误路由给单首帧模型。

## 7. 推荐实施顺序

### P0：先修正错误语义

1. 所有评估失败改为“未评估/不可用”，分数为空。
2. 本地启发式分更名为交付健康度，并与视觉质量完全分离。
3. 生成、评估和人工接受使用三个独立状态，互不覆盖。
4. 统一两套视频评分入口的字段、状态和错误分类。

### P1：降低误杀并提高尾帧达成率

1. 为镜头增加 Hard Exact、Hard Semantic、Soft Directional、Editorial 终态等级。
2. 建立能力驱动路由，Hard Exact 禁止使用不支持尾帧的模型。
3. 上线短格式 HappyHorse Prompt 编译器，加入预算、去重和矛盾检查。
4. 尾帧从单帧绝对匹配改为末尾稳定窗口的核心状态检查。
5. 视觉模型低置信度结果进入灰区，不自动否决。

### P2：建立可持续质量系统

1. 拆分专职评估器并按风险选择执行。
2. 建立黄金集，按镜头类型校准阈值。
3. 多候选采用排序/成对比较，绝对分主要用于解释。
4. 建立模型版本、Prompt 版本和规则版本的离线回归。
5. 以人工推翻率、误杀率和终态达成率驱动迭代。

## 8. 最终建议

对于尾帧问题，最重要的产品和工程原则是：

> Prompt 只能提升软控制的概率，不能把供应商未提供的输入能力变成硬能力。

对于评分问题，最重要的原则是：

> “无法评估”不是“质量为零”，“模型不确定”也不是“必须失败”。

如果只做两件事，优先应是：

1. 用能力路由保证真正需要尾帧的任务进入首尾帧模型，HappyHorse Prompt 只负责可接受的软终态任务。
2. 把评分体系改为带状态和置信度的分层质量决策，彻底取消“调用失败即 0 分”和“所有视觉阈值串联否决”。

这两项会同时减少无效重试、人工审核量和错误的产品预期，也为后续接入新模型、替换视觉评估器和做数据化校准提供稳定边界。
