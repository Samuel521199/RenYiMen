# 一句话成片：功能说明（人话版）

本文面向产品、运营和开发同事，用尽量好懂的语言介绍「一句话成片」是什么、怎么跑、剧本是怎么被拆开的。技术细节以当前代码为准；若与代码不一致，以代码为准并应及时修正本文。

**入口页面**：`/workbench/workflows/one-prompt-video`

---

## 1. 这个功能是干什么的？

用户只需要输入**一句话创意**（例如「30 秒竖屏游戏广告，主角从逆风翻盘到庆祝胜利」）， optionally 再上传几张**参考图**（人物、产品、场景等），系统会自动：

1. 把创意扩写成有因果、有钩子的**广告剧本**；
2. 拆成时间轴上的多个**视频片段**；
3. 生成角色/产品等**一致性参考图**和每个片段的**首尾关键帧**；
4. 调用图生视频模型，把相邻关键帧之间的画面**动起来**；
5. 用 FFmpeg **拼接成完整成片**，上传后可预览下载。

核心理念：**先设计好故事，再生成画面；视频模型只负责「让首尾帧之间顺滑动起来」，不负责发明剧情。**

---

## 2. 用户侧：从头到尾怎么用？

可以把整个体验理解成一条「流水线 + 多个审核关卡」。用户每过一关点一次「批准」，系统才会进入下一步生成。

| 顺序 | 用户在页面上做什么 | 系统在后台做什么 |
|------|-------------------|------------------|
| 1 | 填写一句话、选时长/画幅、上传参考图，创建项目 | 保存到数据库，状态 `DRAFT` |
| 2 | 点击「生成剧本计划」 | 调用大模型做剧本拆解，状态 `PLANNING` → `PLAN_REVIEW` |
| 3 | 审核剧本：看分段、关键帧描述、资产库规划，可改文案 | 用户确认计划 |
| 4 | 批准计划，开始出图 | 先生成**一致性资产图**（人物三视图等），再生成**边界关键帧** |
| 5 | 锁定资产库 | 用户确认角色/产品参考图没问题，系统继续生成剩余关键帧 |
| 6 | 审核全部关键帧图 | 用户看图，不满意可单张重生成 |
| 7 | 批准关键帧 | 系统生成片段内需要的**子分镜参考图**（如有） |
| 8 | 审核子分镜 | 用户确认内部参考图 |
| 9 | 批准子分镜 | 批量提交**图生视频**任务，每段 3–15 秒 |
| 10 | 审核各视频片段 | 用户看每段动效，可单段重生成 |
| 11 | 点击合成 | FFmpeg 拼接、加字幕/转场/BGM，上传 OSS |
| 12 | 预览成片，确认完成 | 状态 `DONE` |

任意生成阶段都可能失败（`FAILED`），支持**取消、恢复、回退**到较早的审核阶段，不必从头再来。

## 4. 项目状态机（系统怎么知道进行到哪一步）

```text
DRAFT          刚创建，还没生成计划
  ↓
PLANNING       正在调用大模型写剧本
  ↓
PLAN_REVIEW    计划写好，等用户审核
  ↓
IMAGE_GENERATING  正在生成图片（资产库 + 关键帧 + 子分镜）
  ↓
IMAGE_REVIEW   边界关键帧出完，等用户看图
  ↓
MICRO_SHOT_REVIEW  子分镜参考图出完，等用户审核
  ↓
CLIP_GENERATING   正在生成各段视频
  ↓
CLIP_REVIEW    片段出完，等用户审核
  ↓
COMPOSING      FFmpeg 合成中
  ↓
FINAL_REVIEW   成片已出，等用户确认
  ↓
DONE           完成

（任一步可能 → FAILED，可 resume / rollback）
```

**五个硬性审核节点**：计划、资产库、关键帧、子分镜、片段——前置产物没准备好，系统不会偷偷跳过。

## 5. 核心数据概念（先搞懂名词）

### 5.1 片段（Segment）

一条视频被切成若干**片段**，每段大约 **3–15 秒**（受上游图生视频模型限制）。  
相邻片段共享一个**边界关键帧**：上一段的最后一帧 = 下一段的第一帧，拼接时才能连贯。

例如 30 秒视频通常会拆成 2–10 段，具体数量由总时长和剧情节奏决定。

### 5.2 关键帧（Keyframe）

**边界关键帧**（编号为正）：时间轴上的「定格画面」，表示某一刻场景长什么样。  
**一致性关键帧 / 资产库**（编号为负）：人物正面、侧面、产品特写等**可复用参考图**，不直接出现在时间轴上，而是给后续出图当「锚点」，防止脸/产品跑偏。

### 5.3 子分镜（Micro Shot）

有的大动作在一个片段内跨度较大，光靠首尾两帧不够，系统会在片段**内部**再拆更细的「子分镜」，必要时生成额外的**中间参考图**，帮助视频模型理解中间过程。不是每个片段都需要。

### 5.4 一致性锚点（Consistency Anchor）

剧本里会出现的人物、产品、道具、场景布局等，系统会为需要锁定的对象建立**锚点**，并决定是否必须出参考图、参考强度多高。后续每次出图都会通过 **Reference Selector** 自动挑选该用哪些已锁定的资产图。

### 5.5 一镜到底（Single Take）

**每个片段内部**要求是一次连续镜头：不能在同一 segment 里偷偷切场景、切机位。如果剧本拆法违反这条规则，**Single-Take Audit** 会拦截并要求重新拆解，而不是硬生成。

跨场景、跨机位的变化应发生在**片段边界**（硬切或转场），而不是 segment 中间。

## 6. 剧本拆解逻辑（重点）

这是整个功能最核心的部分：**用户的一句话，怎么变成可执行的拍摄计划？**

当前默认使用 **v2 三阶段规划器**（环境变量 `ONE_PROMPT_VIDEO_PLANNER_ARCH=v2`），由阿里云通义系列大模型完成，分四步串联（其中第二步又拆成两个子角色）。

### 6.1 总览

```text
用户一句话 + 参考图（可选）
        │
        ▼
┌───────────────────┐
│ ① Planning Architect │  想清楚：这是什么广告？多长？分几段？要哪些资产？
└─────────┬─────────┘
          ▼
┌───────────────────┐
│ ② Storyboard Artist  │  设计剧情：hook、冲突、转折、爽点、CTA
└─────────┬─────────┘
          ▼
┌───────────────────┐
│ ③ Shot Decomposer    │  落地分镜：关键帧、片段、机位、运动合同
└─────────┬─────────┘
          ▼
┌───────────────────┐
│ ④ Prompt Detailer    │  写出生图/生视频的详细提示词
└─────────┬─────────┘
          ▼
   Story Quality Gate（自动质检）
          │
          ├─ 不通过 → 自动重写（有限次数）
          └─ 通过 → 写入 planJson，等待用户审核
```

---

### 6.2 第一步：Planning Architect（规划架构师）

**人话**：先别急着画分镜，先回答「这条片要达成什么广告目标」。

**输入**：
- 用户的一句话创意
- 目标时长（如 30 秒）、画幅（9:16 / 16:9 / 1:1）
- 最多 9 张参考图（若有，会用视觉模型读图，只提取稳定视觉事实，不当成剧情本身）

**输出（写入 planning manifest）**：
- **项目意图**：视频类型、目标观众、成功标准
- **时间轴蓝图（timeline blueprint）**：总共分几段、每段起止秒数、每段的剧情角色（hook / setup / proof / payoff / ending 等）
- **一致性清单**：需要锁定的人物、产品、场景、品牌视觉等锚点
- **风格约束**：色调、光线、负面提示（不要出现什么）
- **初步创意策略**：广告模板类型（游戏反转、产品痛点解决、电商转化等）

**分段数量怎么定？**  
每段最短 3 秒、最长 15 秒。例如 30 秒视频，段数大约在 `ceil(30/15)` 到 `floor(30/3)` 之间，即 2–10 段，模型在此范围内按剧情需要选取。

---

### 6.3 第二步：Storyboard Artist（分镜编剧）

**人话**：在架构师搭好的时间骨架上，写「故事怎么讲才像广告，而不是 PPT 翻页」。

**核心产出**：

#### （1）广告创意策略 `creativeStrategy`

系统会先判断这条片属于哪类广告（游戏、产品、电商、餐饮、剧情短片等），再选用对应**模板**，例如：

| 模板示例 | 典型结构 |
|---------|---------|
| 游戏反转 `game_reversal` | 逆风 → 做选择 → 机制触发 → 爽点释放 → 反应 → 下载 CTA |
| 产品痛点解决 `product_problem_solution` | 痛点 → 使用 → 效果证明 → 品牌记忆 → CTA |
| 电商转化 `ecommerce_offer_conversion` | 痛点 → 卖点证明 → 优惠 → 下单 CTA |

每个模板都要求说清楚：**钩子（hook）、冲突（conflict）、转折点（turning point）、爽点/结果（payoff）、行动号召（CTA）**。

#### （2）剧情节拍 `storyBeats`

把故事拆成若干**节拍**，类似广告脚本里的「第几秒发生什么」：

- 每个 beat 有顺序、情绪、**因果**（cause → effect）
- 复杂动作要求三段式 **actionContinuity**：准备 → 执行 → 反应
- payoff 必须能追溯到前面的证据（`keyEvidenceIds`），不能「突然赢了」

#### （3）镜头分组 `shotGroupingPass`

**相邻的时间轴段落能不能合并成同一个视频片段？** 这里有一套确定性规则（不全靠模型拍脑袋）：

**必须切分的情况**（举例）：
- 合并后超过 15 秒（视频模型单段上限）
- 进入 CTA 段（转化动作要单独成段，不和 payoff 糊在一起）
- payoff 状态突变（「从紧张到庆祝」要切开）
- 时间跳跃、换场景、换空间锚点
- 叙事焦点变了、机位/主客观关系不匹配
- 情绪方向接不上，模型容易生成跳变

**可以合并的情况**：  
同一连续动作链、同一空间、情绪递进自然，且合并后仍 ≤ 15 秒。

这一步的输出会直接影响最终有几个 **Segment**、每个 Segment 承担什么 **storyFunction**（hook / conflict / proof / payoff / cta 等）。

---

### 6.4 第三步：Shot Decomposer（镜头拆解师）

**人话**：把故事翻译成「摄影师能执行的东西」——哪一秒是什么画面、镜头怎么运动、首尾帧分别长什么样。

**主要工作**：

1. **生成边界关键帧列表**  
   每个片段有明确的 start / end 关键帧；时间轴上第 1 帧在 0 秒，最后一帧在总时长秒数。

2. **生成 Segment 列表**  
   每段绑定：起止关键帧编号、时长、视频提示词骨架、机位、主体/环境运动描述、关联的 story beat。

3. **生成 segment render descriptions（执行合同）**  
   每段必须写清：
   - **首帧合同**：场景、人物状态、产品状态
   - **尾帧合同**：这一镜结束时世界变成什么样
   - **运动合同**：镜头和主体怎么动
   - **一镜到底合同**：这一镜内禁止切镜

4. **Camera Graph（机位关系图）**  
   记录机位继承关系，避免相邻片段机位逻辑打架。

5. **Single-Take 修复循环**  
   若审计发现「同一 segment 内偷偷要求切镜」或「动作物理上不可能一镜完成」，会自动让模型**重新拆解**该段，直到通过或达到重试上限。

**长视频优化**：段数较多时，Shot Decomposer 会**按 segment 并行**调用模型，再合并结果，避免一次输出太长导致 JSON 截断。

---

### 6.5 第四步：Prompt Detailer（提示词工程师）

**人话**：前面定好了「拍什么」，这一步写「怎么跟 AI 画图/画视频模型说」。

为以下对象生成中英文提示词和负面约束：
- 每个关键帧的**文生图 prompt**
- 每个片段的**图生视频 prompt**
- 需要的**子分镜参考图 prompt**

同时做 **Reference Selection**：根据当前帧需要的视角（正面/侧面/背面），从已锁定的资产库里**确定性**挑选最合适的参考图，而不是随机堆图。

---

### 6.6 自动质检：Story Quality Gate

计划写完后，系统**不直接交给用户**，先跑一遍规则引擎 `evaluateStoryQualityGate`，检查常见烂剧本：

| 问题 | 含义 |
|------|------|
| `missingHook` | 开头没有抓住注意力 |
| `missingConflict` | 没有痛点/阻力/悬念 |
| `suddenOutcomeRisk` | 结果来得太突然，前面缺少触发和证据 |
| `referenceOveruseRisk` | 只是在「让参考图动起来」，没有重新设计剧情 |
| `ctaTraceMissing` | CTA 和前面的利益点接不上 |
| `payoffReactionMissing` | 爽点之后缺少人物反应 |
| `complexActionContinuityMissing` | 复杂动作缺「准备→执行→反应」 |

**不通过怎么办？**  
系统会自动发起 **Story Quality Rewrite**（有限次数），让模型从指定阶段（创意层或分镜层）重写，直到通过或达到上限。通过后才会进入 `PLAN_REVIEW` 让用户看。

## 7. 批准计划之后：媒体是怎么生成的？

### 7.1 图片阶段（两批顺序）

1. **一致性资产库**（负编号关键帧）：人物三视图、产品图等，优先生成。  
   用户 **approve-assets** 锁定后，这些图才会作为强参考参与后续出图。

2. **边界关键帧**（正编号）：时间轴上的首尾画面。  
   Reference Selector 自动带上已锁定的资产图 + 用户上传的参考图（如有）。

3. **子分镜参考图**（可选）：片段内部需要的中间帧。

**并发**：图片任务默认最多 3 路并行（`ONE_PROMPT_VIDEO_IMAGE_CONCURRENCY`），避免打爆上游配额。

**质量**：可生成多个候选图，经质量评估后推荐最佳；用户也可手动选版本或回退历史版本（最多保留 10 个 media revision）。

### 7.2 视频片段阶段

每个 Segment 使用**首帧图 + 尾帧图**调用图生视频模型（当前默认 `happyhorse-1.1-i2v` 类首尾帧接口），模型只负责两帧之间的运动，**不会**自动补写新剧情。

**并发**：默认最多 2 路并行（`ONE_PROMPT_VIDEO_CLIP_CONCURRENCY`）。

**同步**：前端或后台定期调用 `sync` 接口轮询上游任务状态；完成的片段 URL 写回数据库。

### 7.3 合成阶段

所有片段通过后，**本地 FFmpeg**：
- 按顺序拼接各段 mp4
- 按 `finalTransitionPlan` 加转场（硬切或短 xfade）
- 叠加字幕、BGM，做响度归一化
- 上传 **阿里云 OSS**，得到最终 `finalVideoUrl`

## 8. 修改、回退与脏数据

用户在审核阶段改剧本字段、改提示词、重生成某张图或某段视频时，系统会标记相关产物为 **dirty**（已过期），并沿依赖链传播：

```text
改剧本计划 → 后面的图、片段、成片都可能失效
改某张关键帧 → 依赖它的片段可能失效
改某段视频 → 成片失效
```

支持 **rollback** 回到较早审核阶段（如只回到 `PLAN_REVIEW` 或 `IMAGE_REVIEW`），并清理之后的派生产物，避免「旧图 + 新剧本」混在一起。

## 10. 和「只让参考图动起来」的区别

早期或简化方案容易犯一个错：用户上传了参考图，模型就直接把图拆几段让它动，**没有广告因果**。

当前 v2 流程刻意避免这一点：

1. **参考图只是资产**，不能替代剧情设计（Planning 阶段就写进 prompt 约束）。
2. **必须先有 story beats 和 quality gate**，烂剧本过不了自动质检。
3. **每个 segment 必须声明 storyFunction 和因果**，CTA 必须能追溯到 payoff/proof。
4. **视频模型职责有限**：只执行首尾帧之间的运动，创意必须在规划阶段完成。

## 12. 一句话总结

**一句话成片 = 用大模型把一句创意拆成「有广告逻辑的分段剧本」，再按「资产库 → 关键帧 → 子分镜 → 图生视频 → FFmpeg 合成」流水线生成成片；中间每一步都有审核门禁，剧本拆解强调因果、钩子和一镜到底，避免变成「参考图 PPT 动画」。**

## 13. 三个顽固 Bug 的原因分析与解决方案

这一章的写法和一般排障文档不一样：**不满足于「这个具体案例怎么改好」，而是每个 bug 都往上多问一层——它暴露了架构拆解上的什么问题、工程实现上的什么问题、Prompt 设计上的什么问题**。具体案例（比如「游戏反转类广告」「牛头吉祥物锚点」）只作为例证，不作为解法本身，因为「某个具体场景该怎么写」本来就是大模型能覆盖的事，不需要工程侧针对每个场景单独打补丁。真正需要工程侧解决的，是**架构该怎么拆得让模型更容易做对**、**Prompt 该怎么设计让模型没有偷懒空间**。

每个 bug 按同一个结构展开：

1. **现象**：用户看到的问题。
2. **表面机制**：代码里直接对应的原因（诊断层，仍然需要，但不是终点）。
3. **往上一层：这暴露了什么通病**：脱离这个具体 bug，这类问题在架构/工程/Prompt 设计上普遍会怎么发生。
4. **解决办法：从架构拆分和 Prompt 结构入手**：给的是可以复用到所有场景、所有模板的结构性解法，不是「给某个模板多写一段提示词」。

文末第 13.4 节会把三个 bug 汇总成几条通用设计原则。

---

### 13.1 Bug 1：第一轮剧本拆解特别慢（体感 30 分钟）

**现象**：用户点「生成剧本计划」后，页面转圈，实际观察到接近 30 分钟才出结果，用户体验很差。

#### Bug 1 的表面机制

**（1）这一步在架构上就不是「一个请求」，而是「一串串行请求」，而且是同步阻塞的。**

第 6 章的四个阶段（Planning Architect → Storyboard Artist → Shot Decomposer → Prompt Detailer）不是并行跑的，而是**严格串行**：后一步必须等前一步的 JSON 完全返回才能开始，因为后一步的输入就是前一步的输出。这本身是合理的（剧情要先定，镜头才能落地），但意味着总耗时 = 四次大模型调用耗时之和，而不是取其中最慢的一个。

**（2）串行链路上还藏着好几个「隐形乘数」，平时不容易被注意到：**

- **Shot Decomposer 按 segment 分批调用**：当前默认是「按 segment 拆分别调用」模式（`ONE_PROMPT_VIDEO_SHOT_DECOMPOSER_MODE=segment`），并发上限只有 2（`ONE_PROMPT_VIDEO_SHOT_DECOMPOSER_CONCURRENCY=2`）。一个 30 秒视频常见会拆成 5–8 段，按并发 2 算，就是 3–4 波「排队」调用，每一波都要等最慢的那个 segment 返回。
- **一镜到底修复循环**：Shot Decomposer 出来的结果要过 Single-Take Audit（见 6.4 节），一旦某个片段被判定「需要切镜」或「动作物理上走不通」，系统会**整段重新请求大模型**，最多允许 **4 轮**（初次 + 3 次重试）。这 4 轮里的每一轮都是一次完整的大模型调用，不是快速修补。
- **JSON 自愈重试**：任意一个阶段如果返回的内容解析不出合法 JSON（大模型输出结构复杂的长 JSON 时很常见），系统会自动再发一次「请把这段内容修成合法 JSON」的请求，这又是一次完整的网络往返，且用的是和原请求一样长的超时时间。
- **每次调用的超时上限本身就给到 10 分钟**：默认走流式请求，只要模型还在陆续吐字符（哪怕吐得很慢），空闲超时 90 秒才会掐断，单次调用的硬上限是 **10 分钟**（`ONE_PROMPT_VIDEO_JSON_STAGE_STREAM_MAX_TIMEOUT_MS=600000`）。四个阶段 + 若干次重试，只要其中两三次调用比较慢，累计到 20–30 分钟并不意外。

**（3）整个过程对用户是「一个黑箱」，没有真实进度反馈，体感被进一步放大。**

前端「生成剧本计划」这一步，进度条走的是**假进度**：本质是根据「已经等了多少秒」去套一条提前写好的曲线（`estimatePlanningProgress`），例如 3 秒内是「创建中」，18 秒后跳到「理解创意」，55 秒后跳到「设计分镜」……超过 110 秒后进入一条越来越慢、永远逼近 99.2% 但不到 100% 的对数曲线。这条曲线和后端到底跑到哪一步、还剩几个 segment 没处理**完全没有关系**，纯粹是「猜」。等待越久，这种「进度条在骗我」的感觉越强烈，实际体验比数字上的等待时间更糟。

**（4）请求本身是一次同步 HTTP 调用，容易和网关超时打架。**

`/api/video-projects/{id}/plan` 是一个普通的 `POST`，后端要把上面全部串行流程跑完才会返回响应，中间没有任何分段返回。生产环境经过 Nginx 转发时，`proxy_read_timeout` 设置的是 **600 秒（10 分钟）**，如果后端整个流程真的跑到 20–30 分钟，**网关会先把连接掐断**，前端收到的是超时/网络错误，而不是「模型其实还在算」，用户体感是「卡死了」，而不是「慢一点但在推进」。

#### 往上一层：Bug 1 暴露了什么架构通病

抛开「慢」这个表象，Bug 1 真正反映的是两个可以复用到其他功能上的设计问题：

**通病 A：阶段划分是按「人类分工角色」设计的，不是按「数据依赖」设计的。**  
「Planning Architect → Storyboard Artist → Shot Decomposer → Prompt Detailer」这个四步命名方式很像现实世界里一个广告项目组的分工（策划 → 编剧 → 分镜师 → 文案），这种命名直觉会让人觉得「后一个角色必须等前一个角色完全交完工才能动笔」，就像现实项目管理里的顺序交接一样。但从**数据依赖**的角度看，很多子任务的最小依赖单元比「整个阶段」小得多：比如某个 segment 的 Prompt Detailer 工作，理论上只依赖这一个 segment 自己的 render description，并不需要等其他 segment 的 Shot Decomposer 也跑完。当架构按「角色顺序」而不是「数据依赖图（DAG）」来设计时，会系统性地引入不必要的等待——这是一种**很容易在多阶段 LLM 流水线里重复出现的通病**，不是这一个功能独有的。

**通病 B：可靠性设计堆在「事后补救」层，没有从「减少单次生成复杂度」的源头去设计 Prompt。**  
「JSON 自愈重试」「一镜到底修复循环」都是「模型这次输出不合格，那就重新问一次」的兜底机制。这类机制本身没问题，但如果它经常被触发，说明真正该被检讨的是：**为什么要求模型在一次调用里吐出这么多耦合的结构化内容**（关键帧、segment、render description、camera graph 全部塞进 Shot Decomposer 一次输出）？一次生成承载的结构越复杂，出错概率、返回时长、需要重试的概率都会同步上升。这也是通用规律：**任何要求大模型一次性产出大段强结构化 JSON 的设计，都应该先问「这个产出能不能拆成更小、更独立的子请求」，而不是先假设「模型会大概率一次做对，出错了再重试」。**

这两个通病加在一起，才是「体感 30 分钟」的真正来源：不是模型变慢了，而是**架构把本可以并行、可以拆小的工作，设计成了必须串行、必须整块交付的工作**。

#### Bug 1 的解决办法：从架构拆分和 Prompt 结构入手

**架构层面——把「角色顺序」重新设计成「数据依赖图」：**

- 把四阶段的产物依赖关系画成显式的任务图，而不是隐式的函数调用顺序：标出哪些子任务只依赖「上一阶段的某一小块产出」，哪些子任务真的需要「上一阶段全部产出」。例如某个 segment 的 Prompt Detailer 工作一旦确定只依赖该 segment 自己的 render description，就可以把它从「等 Shot Decomposer 全部 segment 完工才批量进入 Prompt Detailer」，改成「该 segment 出结果就立刻派发」，任务编排层按依赖图调度，而不是按「阶段 1 完 → 阶段 2 完 → 阶段 3 完」的固定顺序。
- 这个思路不止用于 Bug 1：以后任何新增的多阶段 LLM 流程，设计之初就应该先画依赖图、再决定要不要拆成独立子任务，而不是先按「听起来像人类分工」的直觉去命名和串联阶段。

**Prompt 结构层面——把「一次产出很多耦合字段」拆成「多次产出单一职责字段」：**

- 评估 Shot Decomposer 当前一次调用里要求的产出（关键帧列表、segment 列表、render description、camera graph）是否可以按职责拆成更小的独立请求，每个请求的输出 schema 更简单、更容易一次做对，减少「输出格式不对 → 触发 JSON 自愈」「内容不满足一镜到底 → 触发整段重新生成」的概率，而不是靠加大重试上限去覆盖。
- 把「一镜到底」这类硬约束尽量前移到 Prompt 的输出契约设计里（比如强制模型在同一字段里先声明「这一镜是否可能被审计判定为切镜」并给出理由），让模型在第一次生成时就对着更明确的自检项写，比事后靠 Single-Take Audit 打回去重写命中率更高、更省时间。

**基础设施层面——把「进度可观测」做成通用能力，不是这一个功能专属：**

- 任何长耗时、多阶段的任务，都应该有统一的「阶段进度回写」基础设施（当前项目已经在剧本规划任务上补上了这一层，见下方「已落地」说明），后续新增类似的多阶段生成任务时应直接复用这套能力，而不是每个功能各自发明一套进度估算逻辑。

#### Bug 1 已落地的部分与仍待推进的部分

当前实现已经完成「同步改异步 + 真实进度回写」这部分工程改造：

- `POST /api/video-projects/{id}/plan` 只负责把任务置为 `PLANNING` 并入队，立即返回 `202 Accepted`，不再等待四阶段规划全部结束，因此不会再被 10 分钟 Nginx 读取超时截断。
- 后台任务把 `planning_architect`、`storyboard_artist`、`shot_decomposer` 分段完成数、`single_take_audit`、`split_repair`、`json_repair`、`prompt_detailer` 和 `story_quality_gate` 写入 `planJson.plannerProgress`。
- 前端通过现有 `sync` 轮询读取 `plannerProgress`，展示「已完成 N/M 步、剩余 M-N 步」和真实 segment 完成数量，不再用 `estimatePlanningProgress` 猜测 `PLANNING` 状态进度。
- JSON 修复次数/耗时、一镜到底修复次数/耗时会同时写入进度数据和 `project.plan.progress` 日志。
- 规划任务使用持久化 `taskId`、租约和 30 秒心跳防止多实例重复提交；进程退出后租约过期，`sync` / `resume` 会重新接管。
- Planning Architect、Storyboard Artist 和各 Shot Decomposer segment 的成功结果继续使用输入指纹检查点保存；失败恢复只补未完成阶段，不重复消耗已成功阶段。

这项修复解决的是「通病之外」的一个独立问题：同步连接被误杀、黑箱等待、假进度、失败后整链重跑。但上面提到的**通病 A（按数据依赖图重新调度，而不是按角色顺序）**和**通病 B（把大而复杂的单次产出拆成更小的 Prompt 单元）**仍然是待推进的架构性工作——四阶段目前依然是严格顺序执行，Shot Decomposer 依然是一次产出多种耦合结构；这两项做完之后，总耗时才会真正下降，不只是「等待体验变好」。

### 13.2 Bug 2：剧情拆解很扁平，逻辑不闭环（比如「莫名其妙赢了」）

**现象**：用户拿一张游戏参考图想生成游戏广告，结果系统只是让参考图里的画面简单动起来，没有冲突、没有铺垫，角色突然就「赢了」，自己看都觉得逻辑不闭环。

#### Bug 2 的表面机制

**这正是第 6.6 节里 Story Quality Gate 想解决的问题——但当前线上配置下，这道质检门槛实际上处于「摆设」状态。**

代码里质检门禁的行为由三个开关共同决定：

- `ONE_PROMPT_VIDEO_STORY_GATE`：是否评估质量，取值 `off` / `warn` / `strict`。
- `ONE_PROMPT_VIDEO_STORY_REWRITE_MAX`：质检不过时，最多允许自动重写几次，取值 `0` / `1` / `2`。
- 只有 `storyGateMode` 是 `strict` 的时候，质检不通过才会**真正拦住流程**，逼一次重写；`warn` 模式下，质检该该跑还是会跑（会算出 `missingHook`、`suddenOutcomeRisk`、`referenceOveruseRisk` 等问题码），但**只是记录到日志和 `storyQualityReport` 里，不会触发任何重写，也不会阻止项目直接进入 `PLAN_REVIEW` 给用户看**。

而项目当前的默认值（写在 `.env.example` / `.env.local.example`，且本地/线上 `.env` 都没有覆盖）正是：

```text
ONE_PROMPT_VIDEO_STORY_GATE=warn
ONE_PROMPT_VIDEO_STORY_REWRITE_MAX=0
```

翻译成人话：**质检引擎确实在跑，确实能识别出「结果太突然」「只是让参考图动起来」这类问题，但它检测到问题之后什么都不会做，只是悄悄记一笔账，剧本原样放行。** 这就是为什么代码里明明有一整套很精细的 hook/conflict/turning point/payoff/CTA 因果校验（见 6.3、6.6 节），用户实际看到的效果却还是「扁平、无因果」——**校验结果被检测出来了，但从未被用来拦截或改进当前这次生成**。

再往上游看一层：`referenceOveruseRisk` 这个问题码本来就是专门用来抓「只让参考图动起来、没有重新设计剧情」这种情况的（见 6.6 节表格），说明这个具体 bug 场景其实**已经被系统预判到并写了检测规则**，只是检测结果没有被启用来纠正。

#### 往上一层：Bug 2 暴露了什么架构和 Prompt 设计通病

「质检门禁默认是 `warn` + 重写次数 `0`」只是最表层的原因，把这个开关打开只能算「让已经写好的机制生效」，不是「解决了问题的根」。真正值得往上抽的，是三个会在任何「LLM 生成 + 事后质检」类系统里重复出现的通病：

**通病 A：生成和校验是「先生成、再打分、不达标就整体重写」的事后审查架构，硬约束没有前移到生成契约里。**  
当前架构里，「必须有 hook / conflict / turning point / payoff / CTA 因果链」这条要求，只存在于两个地方：一是 Prompt 里的自然语言描述，二是生成完之后的规则引擎打分。这是一种**判别器和生成器分离、且判别器只有软约束力**的结构：模型完全可以在自然语言层面「读到了」这条要求，但因为它不是结构化输出里的必填字段，模型可以在没有该字段的情况下依然产出一份「格式合法但因果薄弱」的 JSON。只要生成器的输出 schema 里没有把「证据链」变成不能省略的字段，无论重写几轮，都只是在「重复同一种可能会跳过硬约束的生成方式」，重写次数只是降低了「蒙混过关」的概率，不是消除了它。

**通病 B：Prompt 里给的是抽象规则，没有具体的「合格 / 不合格」对照示例，模型在有强视觉先验时容易走捷径。**  
「你必须有钩子、冲突、转折」这类指令是**规则型**描述，不是**示例型**描述。当同一次请求里还附带了参考图时，图像内容会提供一个非常具体、非常容易照抄的「捷径答案」（让图里的画面动起来），而文本规则相对抽象、不提供同等具体的「正确答案长什么样」。模型在两种强度不对等的信号之间，很容易向更具体、更省力的图像先验倾斜。这是所有「文本规则 + 强视觉参考」共存的 Prompt 设计里都会遇到的通病：**规则越抽象，越容易被更具体的输入信号盖过。**

**通病 C：「看图」和「编故事」两个认知任务耦合在同一次推理里，输入通道没有做隔离。**  
Planning Architect 阶段是「一句话创意 + 参考图 + 输出剧情结构」一起喂给模型的，模型在同一次推理里既要理解图像，又要设计因果剧情。这种耦合的输入设计，天然给了模型「直接从图像内容里找答案，而不是重新构思剧情」的捷径，且这种偷懒无法通过输出格式校验发现（因为输出格式本身没有问题，只是内容上偷懒了）。

#### Bug 2 的解决办法：从架构拆分和 Prompt 结构入手

**架构层面——把硬约束从「事后打分」前移到「生成契约」里：**

- 把 Story Quality Gate 里已经定义好的问题码（`missingHook`、`suddenOutcomeRisk`、`referenceOveruseRisk` 等）对应的判断依据，转化成 Storyboard Artist 输出 schema 里的**必填结构化字段**，而不是只在事后用独立规则引擎去检查自然语言描述里「有没有体现」这些要素。比如要求模型必须显式填写「这个 payoff 对应哪几个更早的 `keyEvidenceId`」，字段留空或引用不存在的证据直接视为格式不合法（而不是留到质检阶段才发现「结果太突然」）。这样一来，「因果闭环」从「靠模型自觉写好、事后抽查」变成「schema 层面的强制项」，重写机制退化为「格式校验失败后的正常重试」，不再是唯一的质量兜底手段。
- 把 `strict` + 有限次数重写保留作为**兜底层**（用于兜住 schema 约束之外的语义质量问题），而不是唯一防线。这样即使重写次数用尽，最坏情况也是「结构上一定有证据链，但证据链质量一般」，而不是「完全没有证据链」。

**Prompt 结构层面——用「正例/反例对照」替代「抽象规则罗列」：**

- 不针对某一个具体模板（如「游戏反转」）单独加提示词，而是在**所有广告模板的公共提示词层**引入统一的正反例对照结构：给出一段「只是让参考图动起来」的反例剧情骨架和一段「有完整因果链」的正例骨架，让模型对照着两种输出的差异去判断自己在写哪一种，而不是只给一句「不能只让参考图动起来」的抽象禁止。这个结构对所有模板通用，不需要每新增一个广告类型就重新补一遍提示词。
- 把「参考图只提供视觉事实，不能替代剧情」这条约束，从「写在提示词里希望模型遵守」升级成「输入通道层面的强制隔离」：要求模型分两步产出——第一步只允许输出「参考图里有什么客观事实」（人物长相、产品外观、场景布局），不允许在这一步输出任何剧情或因果相关内容；第二步完全基于第一步的事实摘要和用户一句话去设计剧情，且第二步的输入里不再包含原始图像。这样「看图」和「编故事」在数据流上被切成两个独立步骤，模型没有机会在设计剧情的同一次推理里直接照抄图像内容，比一句话的软性提示词约束更可靠。

**验证方式——用回归基线验证的是「结构性约束」，不是「某个模板改得好不好」：**

- 上面两条改完之后，用 `one-prompt-video-script-decomposition-baseline.md` 的固定样本集做回归验证，观察的重点应该是「跨模板的因果字段完整率」和「跨模板的参考图照抄率」这两个通用指标是否整体下降，而不是只盯着某一个具体模板（比如游戏反转）的表现——如果只针对某个模板调好了，换一个新模板大概率会重新暴露同样的问题，说明修的是架构和 Prompt 结构，不是某个模板的参数。

### 13.3 Bug 3：资产库用没用好、分镜有没有好好套用资产，说不清楚

**现象**：系统有「资产库」机制（一致性锚点、参考图锁定），但用户不确定大模型拆分剧本时有没有认真设计这些固定资产，后续关键帧/分镜生成时有没有真的用上，最终成片给人的感觉就是「劣质」。

#### Bug 3 的表面机制

资产库的**执行链路本身是完整且经过设计的**：

```text
Planning Architect 定义一致性锚点（consistencyManifest.anchors）
  → Storyboard Artist / Shot Decomposer 给每个关键帧/片段/子分镜
    标注「这一帧用到哪些锚点」（usesConsistencyAnchors / requiredAnchorIds）
  → 生成图片/视频时，Reference Selector 按「这一帧声明要用的锚点」
    去资产库里挑最合适的已锁定参考图
  → 生成结果出来后，视觉质量评估会拿「被选中的参考图」和生成结果做比对，
    打一个 identityScore（身份一致性分）
```

问题出在**第二步和第三、四步之间的耦合方式**：Reference Selector 挑图、身份一致性打分**完全依赖第二步里大模型有没有老老实实在每个关键帧/片段/子分镜上填好 `usesConsistencyAnchors` 这个字段**。系统里目前只有两类校验：

1. **存在性校验**：如果某个关键帧/片段引用了一个「资产库里根本没有」的锚点 ID，会报 `MISSING_ANCHOR_REFERENCE`——这只能抓「写错了 ID」，抓不了「压根没写」。
2. **硬锚点必选校验**：如果某个关键帧/片段**已经声明**要用某个「hard」级别的锚点，但这个锚点最终没有被选中或还没批准参考图，会报 `HARD_ANCHOR_IMAGE_MISSING` / `HARD_ANCHOR_NOT_SELECTED`——这只在「模型已经声明要用」的前提下才会触发。

**换句话说：如果大模型在 Shot Decomposer 阶段偷懒，某个关键帧明明画面里应该出现主角/产品，但它忘了在 `usesConsistencyAnchors` 里把主角/产品的锚点 ID 填上去，当前系统里没有任何一层会发现这个疏漏。** 后续生成这一帧图片时：

- Reference Selector 拿到的「必须使用的锚点」列表是空的，于是它可能只会挑一些「软」参考（比如上一帧的时间轴延续证据），而不会强制带上资产库里锁定的角色/产品参考图；
- 生成出来的画面自然容易跑偏（脸型变了、产品变了样）；
- 到了视觉质量评估这一步，`identityScore` 的评分逻辑是「拿生成结果和 `selectedReferenceUrls`（被选中的参考图）比对」——但如果上一步就没选中任何强参考图，这个分数从一开始就没有比对对象，**质量评估本身也失去了发现问题的依据**，很可能给出一个虚高的分数放行。

这也解释了为什么这个 bug 和 Bug 2 经常一起出现：**同一次「大模型偷懒/拆解不细致」的调用，既会导致剧情扁平（Bug 2），也会同时导致锚点标注缺失（Bug 3）**——本质上是同一个「LLM 输出质量」问题在两个不同维度上的表现，而当前架构里，防止这两种偷懒的机制**都严重依赖模型自己在结构化字段里主动、正确地填好内容**，系统缺少一层「不管模型有没有主动说，我都能自己判断这一帧该不该用哪个锚点」的独立校验。

#### 往上一层：Bug 3 暴露了什么架构和 Prompt 设计通病

「锚点标注是模型自愿填写、校验只查错不查漏」是表层原因。往上抽一层，这里暴露的是三个会在任何「让 LLM 自己声明依赖关系」的系统里重复出现的通病：

**通病 A：把「内容生成」和「依赖声明」这两件事，压缩进同一次生成里，且依赖声明不是模型的注意力焦点。**  
Shot Decomposer 在一次调用里既要写好画面内容（场景、人物状态、运动），又要顺手把「这一帧用到了哪些资产锚点」标注出来。对模型来说，前者是「主任务」（写得好不好直接体现在输出的画面描述质量上），后者是「附加的簿记任务」（标或不标，短期内不影响画面描述本身看起来好不好）。任何设计成「主任务 + 顺手附加一个簿记字段」的 Prompt，附加字段被漏填的概率天然更高——这不是模型不够聪明，而是任务结构本身把「簿记」放在了模型不太在意的位置。

**通病 B：依赖关系的可靠来源，本可以从更早、更受模型重视的结构化字段里**确定性推导**，但架构选择了「再问模型一遍」。**  
Storyboard Artist 阶段的 `storyBeats` 已经要求模型认真填写「这个节拍涉及谁、涉及什么产品」（这是模型会认真对待的核心叙事字段，因为它直接决定剧情是否合理）。但「这一帧用到哪些锚点」这个信息，理论上可以从这些已经认真填写的叙事字段**反推**出来，架构上却选择了在下游阶段让模型「再自愿声明一次」，相当于把一个可以通过确定性推导获得的信息，又交给了模型的自由发挥，多了一次不必要的、可靠性更低的信息传递环节。

**通病 C：校验体系只做「声明一致性校验」（declare-and-verify 里的 verify 部分），没有「声明覆盖率校验」。**  
现有两条校验规则（存在性校验、硬锚点必选校验）都建立在「模型已经声明了什么」的基础上做二次确认，本质上是在检查「说出来的话有没有错」，但完全没有能力检查「该说的话有没有漏说」。这是所有「先让 LLM 自由声明依赖，再做校验」的系统都容易踩的坑：**校验逻辑天然继承了声明阶段的盲区**，因为校验的输入本身就来自那个可能不完整的声明。

#### Bug 3 的解决办法：从架构拆分和 Prompt 结构入手

**架构层面——减少「模型自愿标注」的责任面，改成从更可靠的结构化信息里做确定性推导：**

- 把「这一帧该用哪些锚点」的第一版结果，改成由系统在 Storyboard Artist 产出 `storyBeats`（人物/产品/场景的叙事归属）之后，用确定性规则**自动推导**出一份初始的锚点引用关系（例如「这个 beat 的涉及对象里出现了主角，主角锚点就默认关联到这个 beat 覆盖的所有关键帧/片段」），再把这份初始结果作为 Shot Decomposer 阶段的默认值，模型只需要在有特殊情况时**修改**默认值，而不是从空白状态**创造**标注。让模型承担的是「校对」而不是「凭空填写」，这是可靠性更高的任务设计方式，同样的思路可以推广到所有「下游阶段依赖上游已经确定的结构化信息」的场景，不止用在锚点标注上。
- 把「资产锚点的应用责任」尽量往「锚点定义」这个源头收拢，缩短「定义」和「声明使用」之间要经过的阶段数（当前是 Planning Architect 定义 → Storyboard Artist / Shot Decomposer 分别声明使用，中间隔了两层传递，每一层都有信息衰减的机会）。

**校验层面——补上「覆盖率校验」这一类独立于模型自我声明的检查：**

- 在 Plan Validator 里新增一类检查，和现有的「存在性/硬锚点必选」校验形成互补：现有校验管「不要引用错的」，新校验管「不要漏引用该用的」——对每个标记为必须保持一致的锚点，检查它在时间轴上的引用次数是否达到与其重要程度相符的下限。这类「反向覆盖率检查」应该成为项目里所有「LLM 自由声明依赖关系」模块的标准配置，而不是只给资产锚点单独补一次。
- 视觉质量评估在「本该有强制引用锚点却没有可比对参考图」的情况下，不应该正常打分，而应该标记为不可信/待复核——评估逻辑的可信前提是「比对对象存在」，比对对象缺失时给出的分数没有意义，这一条同样适用于任何「以参考图/参考对象为基准做评分」的模块。

**Prompt 结构层面——让「依赖声明」变成模型必须认真处理的显式产出，而不是隐藏在长 JSON 里的次要字段：**

- 把「锚点引用」从 Shot Decomposer 输出 JSON 里的一个普通字段，提升为一个需要模型先给出理由再给出结论的显式步骤（例如要求模型先用一句话说明「这一帧画面里出现了谁/什么产品」，再据此填写锚点 ID），用「先说理由再给结论」的结构逼模型认真过一遍画面里到底出现了什么，而不是让它在长长的 JSON 里顺手填一个容易被忽略的数组字段。这个结构性技巧同样可以用在其他「容易被模型当作次要字段敷衍」的标注类产出上。

### 13.4 三个 bug 背后的共同结论：四条可复用的设计原则

把三个 bug 的「往上一层」分析放在一起看，表层原因各不相同（延迟、剧情扁平、资产漏标），但抽出来的架构/工程/Prompt 通病高度重叠，可以归纳成四条原则，建议作为以后设计任何新的多阶段 LLM 功能时的检查清单，而不只是这一个功能的修复项：

**原则一：阶段划分要按「数据依赖图」设计，不要按「听起来像人类分工」的顺序设计。**  
（对应 Bug 1 通病 A）四阶段命名成「架构师 → 编剧 → 分镜师 → 文案」很直觉，但直觉命名容易诱导出「必须顺序交接」的错误架构决策。正确做法是先画出每个子任务真正依赖哪些更小粒度的上游产物，再决定哪些工作可以并行、可以提前派发。

**原则二：模型一次生成承载的结构越复杂，出错和变慢的概率越高——该拆的产出要拆成独立的小请求，而不是靠加大重试次数兜底。**  
（对应 Bug 1 通病 B）「反正出错了可以重试」是一种事后补救思维，治标不治本；更省时间、更稳定的做法是从源头降低单次生成的复杂度。

**原则三：真正重要的硬约束，要写进生成阶段的输出契约（schema 必填字段），不能只停留在「事后质检 + 不达标重写」这一层，也不能只停留在抽象的自然语言规则里——尤其是当输入里还有更具体的视觉参考图时，抽象规则很容易被具体的图像先验盖过。**  
（对应 Bug 2 通病 A、B、C）质检和重写应该是兜底层，不该是唯一防线；Prompt 里给正反例对照，比给抽象规则描述更有约束力；容易被图像内容主导的认知任务（看图 vs 编故事），应该在输入通道层面强制拆开，不能只靠一句提示词约束。

**原则四：任何「让模型自己声明依赖关系/自己打标签」的设计，都要有独立于模型自我声明的覆盖率校验，并且优先考虑能否从更可靠的结构化信息里确定性推导出这份声明，而不是让模型从空白状态凭空填写。**  
（对应 Bug 3 通病 A、B、C）自由声明 + 事后校验的组合天然只能抓「错」，抓不了「漏」；能靠确定性推导拿到的信息，不要重新交还给模型的自由发挥。

**关于优先级**：Bug 1 的「同步改异步、真实进度回写」已经按短期工程改造落地（见 13.1 节末尾），这解决的是用户体验层面最痛的部分，但原则一、原则二对应的架构拆分（按依赖图调度、拆小 Prompt 产出）仍未开始，总耗时还有下降空间；Bug 2、Bug 3 对应的原则三、原则四目前还停留在「机制已写好但没启用/没补全」的阶段，`strict` 配置开关仍然是成本最低、见效最快的第一步，但要清楚它只是让现有软约束生效，不能替代 schema 层面的硬约束前移和覆盖率校验这两项结构性工作——**这四条原则做完之后，才算是从「治当下这个具体案例」上升到「以后新模板、新场景默认就不容易再犯同一类错」。**
